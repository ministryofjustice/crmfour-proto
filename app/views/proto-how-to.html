Variables

Variables are used in this prototype to store, remember and display data on the page. They can be seen in the URL and look like this: AppStatus=Rejected

Variables can be added to any URL to see how it changes the page. They are always appended to the end of the page name using a single '?' e.g page1?variableName=value

The ampersand '&' symbol can be used to apply multiple variables to a page, for example: page1?cwAppStatus=Rejected&Appeal=Yes

Within a page, variables are used to display different values or content according to the current status - this means not need to separate pages for version of a page.

Forms and displaying of user entered data

The majority of all forms in this prototype allow users to enter data and see it repeated back to them in pages like the 'Check your answers' page. This gives a sense of a real system.

When displaying data on a page you can provide an alternative if there is no value available: 'or': {{data['userName'] or "John Smith"}}

Key variables used in this prototype include:

Caseworker: cwAppStatus

Used to define the current state of an application or claim for caseworkers.

  • cwAppStatus=New : an unassigned new app/claim
  • cwAppStatus=In progress : an open app/ claim that is assigned to a CW and in progress
  • cwAppStatus=Referred : an open app/claim that was referred to the ICA
  • cwAppStatus=Review : an open app/claim that has been sent back to the provider for updates
  • cwAppStatus=Returned : an open app/claim that has been returned by the provider with updates
  • cwAppStatus=Hold : an open app/claim that is waiting to be sent to the ICA
  • cwAppStatus=Part granted : a closed app/claim that was part granted
  • cwAppStatus=Granted : a closed app/claim that was granted
  • cwAppStatus=Rejected : a closed app/claim that was rejected
  • cwAppStatus=Expired : a closed app/claim where the CW requested an update but the provider never responded

Provider: app-task

Used to define the current state of an application or claim for providers.

  • app-task=Referred : an open app/claim that was referred to the ICA
  • app-task=Review : an open app/claim that has been sent to the provider for updates
  • app-task=Submitted : a 'Submitted' app/claim or appeal that has submitted by the provider
  • app-task=Returned : a 'Resubmitted' app/claim that has been returned by the provider with updates
  • app-task=Part granted : a closed app/claim that was part granted
  • app-task=Granted : a closed app/claim that was granted
  • app-task=Rejected : a closed app/claim that was rejected
  • app-task=Expired : a closed app/claim where the CW requested an update but the provider never responded

icaDecision

Has the ICA been sent the claim or application and / or made a decision. For example, using the following combination would mean that the ICA has requested more information from the provider: cwAppStatus=Review&icaDecision=Yes

  • icaDecision=Yes : the decision on the app/ claim was made by the ICA
  • icaDecision=No : the decision on the app/ claim was not made by the ICA

Appeal

Used to display appeal information on the aplication or claim.

  • Appeal=Yes
  • Appeal=Yes2 : used on provider when there is more than one appeal
  • Appeal=No

PA - toggling costs on/off

  • addAdd=0 : No additional costs
  • addAdd=1 : 1 additional cost
  • addAdd=2 : 2 additional costs
  • addTravel=No : No travel costs
  • addTravel=Yes : There are travel costs

Payments - payment type

Used to display the correct form fields for the claim type when making a payment request.

  • paymentType=Non-standard magistrates'
  • paymentType=Non-standard magistrates' - supplemental
  • paymentType=Non-standard magistrates' - appeal
  • paymentType=Non-standard magistrates' - amendment
  • paymentType=Assigned counsel
  • paymentType=Assigned counsel - appeal
  • paymentType=Assigned counsel - amendment

Payments - linked to the original CRM7 or CRM8

Used to display the correct form fields and costs when making a payment request. Data can pull through and some fields can be pre-populated if there is a linked claim.

  • linkedCRM=No : No linked CRM
  • linkedCRM=Yes : Has linked CRM

Versioning

When significant pieces of functionality go live it is often good to create a new version of the PA or NSM respectively. This lets you track how things have changed over time. However, there is not a defined time to do it.

Version PA and NSM separately as the changes tend to be service specific.

  • On the index page put latest service links under 'Previous versions' and change all of the url's to the new version.
  • In the files create a copy of the latest folder and rename it to the new version
  • All of the files in that folder need the follow code changed: {% raw %}{% set version = "latest-pa" %} or {% set version = "latest-nsm" %}{% endraw %} to be the new version number - this must be exactly as you have named the new folder
  • In the folder 'routes' (just above 'Views') create a new routes file based on latest and name it the new version. In the file itself there are 2 references to the version, update these
  • In the js page routes.js (just below this page) add in the new route js file that you created in the previous step e.g. require('./routes/nsm-1-3.js')

When using routes for conditional questions, like radio's buttons, the form action needs to have {% raw %}{{version}}{% endraw %} included so it will use the correct version as specfied in the page: {% raw %}action="/{{version}}/name-of-route" method="post"{% endraw %}
